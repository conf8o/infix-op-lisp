# トップレベル宣言（Decl）

トップレベルに置ける「宣言（Decl）」を導入し、構文フェーズ・型検査フェーズでの環境更新の扱いを整理する。

## ステータス

ARCHIVED

- 2026-03-20 開始
- 2026-03-21 ARCHIVED. トランスパイル処理の流れは `ADR/transpilation-pipeline.md` に分離。

## 背景

現状検討しているトップレベルの宣言には、少なくとも次がある。

- **トップレベルの型宣言**（型注釈・型シグネチャ）
  - 例: `(x : Int)` / `(f : (Int -> Int -> Int))`
- **中置演算子の宣言**（中置の糖衣構文の有効化）
  - 例: `(infix non ==)` / `(infix right ->)`

どちらも「式の評価」そのものではなく、以降の処理（構文の脱糖・型検査など）に影響する。
ただし、影響するレイヤーは同一ではない。

- 中置演算子の宣言は、**構文（脱糖）環境**を更新する
- トップレベル型宣言は、**型環境（シグネチャ環境）**を更新する

この差を曖昧にしたままDeclを増やすと、
「どの宣言がどのフェーズで効くか」「順序は意味を持つか」「型検査でどう扱うか」が散逸しやすい。

本ADRは、具体的な実装は保留しつつ、
Declのインターフェース・環境データ構造・性質・未決事項を整理する。

## ゴール

- トップレベルにおけるDeclを、**“環境更新”として一段抽象化**して扱えるようにする
- Declが影響するフェーズ（構文/型検査）と、適用タイミング（順序依存の有無）を明確化する
- 型検査での挙動（宣言と定義の整合性チェック、未定義の扱い、重複の扱い）を議論できる状態にする
- 将来のDecl（型定義、モジュール、import等）を追加できる拡張点を残す

## 非ゴール

- パーサ・型検査器・トランスパイラの具体的な実装詳細
- 局所（let/fn内部）での中置演算子宣言の導入
- モジュールシステム/別ファイル参照の仕様確定

## 決定事項（暫定）

- `infix` 宣言とトップレベル型宣言は、トップレベルの走査により環境を積み上げ、**後続のフォームに対して有効**とする（順序依存）
- 同名の宣言（`infix` の同一演算子、型宣言の同一識別子）の再宣言はエラーとする

帰結:

- `infix` と型宣言は、**それを利用するフォームより前**に書く必要がある（前方参照はしない）

## 取り扱う対象（暫定）

### 1) 中置演算子の宣言

```clojure
(infix right ->)
(infix right ::)
(infix left  |>)
(infix non   ==)
```

期待する性質（整理）:

- 1つの括弧内では同じ演算子を連鎖できる（left/rightの場合）
- non の場合は連鎖できない
- 連鎖の脱糖は「二項演算のネストによる正規形」になる

### 2) トップレベルの型宣言

```clojure
(x : Int)
(def x 10)

(f : (Int -> Int -> Int))
(def (f x y) (+ x y))
```

期待する性質（整理）:

- 型宣言は “x を Int として扱う” という **型シグネチャの提示**
- 型検査は、定義（def）側の型が宣言と整合することを確認できる

### 形式（インターフェース）案

Declを、更新対象の環境を明示した形で表せるようにする。

- **案A: Declを種類で分け、フェーズごとの処理器で扱う**
  - Decl自体は単純な直和型（infix/type-sigなど）
  - 「構文フェーズ処理」「型検査フェーズ処理」がそれぞれ対応するDeclだけを読む

このADRでは、議論のために **案A（種類で分ける）** を基本線として扱う。

## 環境（データ構造）案と性質

### FixityEnv（中置演算子テーブル）

保持したい情報（例）:

- 演算子名（シンボル）
- 結合方向: left / right / non
- 連鎖可否: non のとき不可、left/right のとき可

データ構造案:

- **連想リスト**（`(op * info) list`）
  - 性質: 追加が簡単
  - 懸念: 参照がO(n)、再宣言エラーの検出も線形探索になりやすい

最低限ほしい性質:

- 決定性: 同じ入力に対して同じ脱糖結果
- 効果範囲: 本ADRの暫定決定では後続のフォームに有効（順序依存）

### TypeEnv（トップレベル型シグネチャ）

保持したい情報（例）:

- 値/関数名（識別子）
- 型（型式）
- （将来）多相性: 型スキーム（`forall`）

データ構造案:

- **連想リスト**
  - 性質: 追加が簡単、スコープ拡張（先頭追加）と相性が良い
  - トップレベルでは重複検出がしづらい（再宣言エラーにしたい場合は追加の検査が必要）

最低限ほしい性質:

- 宣言と定義の整合性検査ができる（後述）
- 型推論を導入する場合でも拡張できる（将来: 型変数、制約、単一化など）

## 決めるべきこと（チェックリスト）

このADRの目的は、ここを埋めていくための論点表を作ること。

1. Declはトップレベルだけか（現段階はトップレベルのみで良いか）
2. `infix` 宣言の効果範囲（後続のみ）【暫定決定】
3. `infix` の重複宣言の扱い（エラー）【暫定決定】
4. 型宣言の効果範囲（後続のみ）【暫定決定】
5. 型宣言と定義の整合性条件（等価 / 片方向 / 包摂）
6. 宣言のみで定義なしを許すか（将来のimportやモジュールの前提）
7. トップレベルの順序（宣言は順序依存: 暫定決定。前方参照は要検討）
8. 予約された演算子（例: `:` `->`）をユーザー宣言で上書きできるか
9. 変数スコープ（トップレベル識別子）とTypeEnvのキー設計

## 未決定事項

- トップレベルが逐次評価モデルか、全体を一括で束縛するモデルか
- `def` の集合を `let rec ... and ...` のような相互再帰にできるか（対象を関数に限定するか等）
- 型宣言の多相性（`forall`）を将来的にどう導入するか
- 型式のパースにおける `->` の扱い（通常の中置脱糖の再利用か、型専用の構文として別処理にするか）
