# トップレベル宣言（Decl）

トップレベルに置ける「宣言（Decl）」を導入し、構文フェーズ・型検査フェーズでの環境更新の扱いを整理する。

## ステータス

IN DISCUSSION

- 2026-03-20 開始

## 背景

現状検討しているトップレベルの宣言には、少なくとも次がある。

- **トップレベルの型宣言**（型注釈・型シグネチャ）
  - 例: `(x : Int)` / `(f : (Int -> Int -> Int))`
- **中置演算子の宣言**（中置の糖衣構文の有効化）
  - 例: `(infix non ==)` / `(infix right ->)`

どちらも「式の評価」そのものではなく、以降の処理（構文の脱糖・型検査など）に影響する。
ただし、影響するレイヤーは同一ではない。

- 中置演算子の宣言は、**構文（脱糖）環境**を更新する
- トップレベル型宣言は、**型環境（シグネチャ環境）**を更新する

この差を曖昧にしたままDeclを増やすと、
「どの宣言がどのフェーズで効くか」「順序は意味を持つか」「型検査でどう扱うか」が散逸しやすい。

本ADRは、具体的な実装は保留しつつ、
Declのインターフェース・環境データ構造・性質・未決事項を整理する。

## ゴール

- トップレベルにおけるDeclを、**“環境更新”として一段抽象化**して扱えるようにする
- Declが影響するフェーズ（構文/型検査）と、適用タイミング（順序依存の有無）を明確化する
- 型検査での挙動（宣言と定義の整合性チェック、未定義の扱い、重複の扱い）を議論できる状態にする
- 将来のDecl（型定義、モジュール、import等）を追加できる拡張点を残す

## 非ゴール

- パーサ・型検査器・トランスパイラの具体的な実装詳細
- 局所（let/fn内部）での中置演算子宣言の導入
- モジュールシステム/別ファイル参照の仕様確定

## 決定事項（暫定）

- `infix` 宣言とトップレベル型宣言は、トップレベルの「宣言フェーズ」で事前収集し、ファイル全体に対して有効とする（順序非依存）
- 同名の宣言（`infix` の同一演算子、型宣言の同一識別子）の再宣言はエラーとする

## 取り扱う対象（暫定）

### 1) 中置演算子の宣言

```clojure
(infix right ->)
(infix right ::)
(infix left  |>)
(infix non   ==)
```

期待する性質（整理）:

- 1つの括弧内では同じ演算子を連鎖できる（left/rightの場合）
- non の場合は連鎖できない
- 連鎖の脱糖は「二項演算のネストによる正規形」になる

### 2) トップレベルの型宣言

```clojure
(x : Int)
(def x 10)

(f : (Int -> Int -> Int))
(def (f x y) (+ x y))
```

期待する性質（整理）:

- 型宣言は “x を Int として扱う” という **型シグネチャの提示**
- 型検査は、定義（def）側の型が宣言と整合することを確認できる

## 中核アイデア: Decl を「フェーズ付きの環境更新」として扱う

トップレベルプログラムを「項目の列」とみなし、項目には少なくとも次を含める。

- Decl（宣言）
- Def（値/関数の定義）
- Expr（トップレベル式。将来的にREPLやスクリプト用途で必要）

Declは “何かを計算する” のではなく、**以降の解釈・検査のための環境を更新する**。
ただし、環境には複数のレイヤーがある。

- 構文環境（SyntaxEnv）: 中置演算子テーブルなど
- 型環境（TypeEnv）: 変数/関数の型（型スキーム）など
- （将来）型定義環境（TypeDefEnv）: 型構成子・コンストラクタなど

### 形式（インターフェース）案

Declを、更新対象の環境を明示した形で表せるようにする。

- **案A: Declを種類で分け、フェーズごとの処理器で扱う**
  - Decl自体は単純な直和型（infix/type-sigなど）
  - 「構文フェーズ処理」「型検査フェーズ処理」がそれぞれ対応するDeclだけを読む

- **案B: Declを“効果（delta）”として持つ**
  - Declが `SyntaxEnv -> SyntaxEnv` や `TypeEnv -> TypeEnv` のような更新関数（差分）を内包する
  - ただし実装寄りに傾きやすいので、現段階では“概念”として扱う

このADRでは、議論のために **案A（種類で分ける）** を基本線として扱う。

## 環境（データ構造）案と性質

### SyntaxEnv（中置演算子テーブル）

保持したい情報（例）:

- 演算子名（シンボル）
- 結合方向: left / right / non
- 連鎖可否: non のとき不可、left/right のとき可

データ構造案:

- **連想リスト**（`(op * info) list`）
  - 性質: 追加が簡単
  - 懸念: 参照がO(n)、再宣言エラーの検出も線形探索になりやすい
- **永続Map**（キー: 演算子シンボル）
  - 性質: 参照がO(log n)、重複検出が容易（再宣言をエラーにする前提と相性が良い）
  - 懸念: 将来“シャドーイング（上書き）”を許したくなった場合、別仕様が必要

最低限ほしい性質:

- 決定性: 同じ入力に対して同じ脱糖結果
- 効果範囲: 本ADRの暫定決定ではファイル全体に有効（宣言フェーズで事前収集）

### TypeEnv（トップレベル型シグネチャ）

保持したい情報（例）:

- 値/関数名（識別子）
- 型（型式）
- （将来）多相性: 型スキーム（`forall`）

データ構造案:

- **連想リスト**
  - 性質: 追加が簡単、スコープ拡張（先頭追加）と相性が良い
  - トップレベルでは重複検出がしづらい（再宣言エラーにしたい場合は追加の検査が必要）
- **永続Map**
  - 性質: 重複検出が容易、参照が安定
  - 懸念: “再宣言”を許すかどうかの仕様が必要（本ADRの暫定決定では許さない）

最低限ほしい性質:

- 宣言と定義の整合性検査ができる（後述）
- 型推論を導入する場合でも拡張できる（将来: 型変数、制約、単一化など）

## 型検査時の挙動（論点整理）

トップレベルの型宣言を導入すると、少なくとも次のケースを仕様化する必要がある。

### 1) 宣言あり + 定義あり

```clojure
(f : (Int -> Int -> Int))
(def (f x y) (+ x y))
```

論点:

- 宣言は「注釈（annotation）」か「シグネチャ（signature）」か
  - 注釈: “推論された型”が宣言と一致することを確認
  - シグネチャ: “宣言された型”を前提に本体をチェック
- 不一致の扱い
  - エラーにする（基本）
  - ある種の“包摂（subtyping）”を許すか（現段階では不要）

### 2) 宣言あり + 定義なし

```clojure
(x : Int)
; def x が存在しない
```

論点:

- エラーにする（未定義宣言）
- 将来のモジュール/外部参照を見越して許す（interface的な扱い）

### 3) 宣言なし + 定義あり

```clojure
(def (f x y) (+ x y))
```

論点:

- 従来通り推論/型付けして型を得る
- 得られた型をトップレベルTypeEnvに登録する

### 4) 重複宣言・重複定義

```clojure
(x : Int)
(x : Bool)

(def x 10)
(def x true)
```

論点:

- **重複宣言はエラー**（暫定決定）
- 重複定義の扱い（エラー / “最後に出たものが勝つ” など）

### 5) 宣言と順序（前方参照）

宣言が「以降に効く」のか、「ファイル全体に効く」のかで設計が大きく変わる。

- **順序非依存（事前収集）**（暫定採用: 宣言フェーズ）
  - 先に全宣言を集めて TypeEnv を構築し、その後に定義を検査
  - 前方参照（宣言が先・定義が後）と相性が良い
  - 「宣言を途中で差し替える」ような逐次的な挙動は表現しない

- （参考）順序依存（逐次）
  - `(x : Int)` は以降の範囲でのみ有効
  - 直観的だが、前方参照（後で定義される名前）と衝突しやすい

## 中置演算子宣言の適用タイミング（論点整理）

中置演算子は構文の脱糖に関わるため、適用タイミングが重要。

### 論点: 宣言の効果範囲

- **順序非依存（事前収集）**（暫定採用: 宣言フェーズ）
  - 先に全 `infix` を集めてから脱糖
  - ただし、ソースの途中に宣言を書く意味は薄れる

- （参考）順序依存（後続のみ）
  - `(infix left |>)` が出た後に現れる式だけが `|>` を中置として解釈する
  - 「宣言」という語感に近い

### 論点: “infixと判定する条件”

`(a op b)` を見たときに、opが未宣言ならどうするか。

- **宣言済みのopだけを中置として脱糖**（未宣言は通常のリストとして扱う）
- **ある種の予約語は常に中置扱い**（例: `:` や `->`）
- **未宣言の中置っぽい形はエラー**（教育的だが、柔軟性を失う）

## アプローチ案（処理パイプライン）

ここでは「S式の読み取り」と「Declの適用」を分離することを前提に、いくつかのパイプライン案を列挙する。

### 案1: 逐次（1パス）

- 入力をS式列として読む（宣言に依存しない）
- 上から順に走査してDeclを適用しつつ、式を脱糖・型検査する

利点:

- “宣言が後続に効く” を素直に表現できる
- エラーロケーションが自然

欠点:

- 前方参照が困難になりやすい
- 「型宣言は先に書く必要がある」などの制約が増える可能性

### 案2: フェーズ分離（複数パス）

- パスA: infix宣言だけ事前収集 → SyntaxEnv を確定
- パスB: SyntaxEnv を使って全体を脱糖（正規形ASTへ）
- パスC: 型宣言だけ事前収集 → TypeEnv（シグネチャ）を確定
- パスD: 定義を型検査（シグネチャ整合性もチェック）

利点:

- 構文と型の依存関係を分離できる
- 型宣言の前方参照を許しやすい

欠点:

- “途中から宣言が効く” という逐次性を失いやすい
- 仕様として「infix/type宣言は全体に効く」と割り切る必要が出る

本ADRの暫定決定（宣言フェーズで事前収集・順序非依存）と整合するため、
当面はこの案をベースに議論する。

### 案3: 宣言の配置ルールで単純化

- `infix` 宣言はファイル先頭にのみ置ける（以降全体に有効）
- 型宣言も（ある範囲）先頭に置ける、または `def` 直前に限定する

利点:

- 実装も仕様も単純になりやすい
- エラーも分かりやすい

欠点:

- 表現力が下がる
- “宣言を途中で挿入する” という書き方ができない

## 決めるべきこと（チェックリスト）

このADRの目的は、ここを埋めていくための論点表を作ること。

1. Declはトップレベルだけか（現段階はトップレベルのみで良いか）
2. `infix` 宣言の効果範囲（全体: 宣言フェーズで事前収集）【暫定決定】
3. `infix` の重複宣言の扱い（エラー）【暫定決定】
4. 型宣言の効果範囲（全体: 宣言フェーズで事前収集）【暫定決定】
5. 型宣言と定義の整合性条件（等価 / 片方向 / 包摂）
6. 宣言のみで定義なしを許すか（将来のimportやモジュールの前提）
7. トップレベルの順序（宣言は順序非依存: 暫定決定。定義の前方参照は要検討）
8. 予約された演算子（例: `:` `->`）をユーザー宣言で上書きできるか
9. 変数スコープ（トップレベル識別子）とTypeEnvのキー設計

## 未決定事項

- トップレベルが逐次評価モデルか、全体を一括で束縛するモデルか
- `def` の集合を `let rec ... and ...` のような相互再帰にできるか（対象を関数に限定するか等）
- 型宣言の多相性（`forall`）を将来的にどう導入するか
- 型式のパースにおける `->` の扱い（通常の中置脱糖の再利用か、型専用の構文として別処理にするか）
