# 変数のスコープと識別子

変数のスコープ管理と識別子の設計について定義します。

## ステータス

IN DISCUSSION

- 2026-03-08 開始

## 背景

変数は名前だけでは一意に識別できません。スコープの概念を導入し、同じ名前でも異なるスコープの変数を区別できるようにする必要があります。

## 決定事項

### 1. 変数はスコープ識別子を持つ

変数は「名前」と「スコープ識別子」のペアで表現します。

```ocaml
type scope_identifier = string
type var = string * scope_identifier
```

**理由**:
- 同じ名前の変数でも、異なるスコープであれば別の変数として扱える
- シャドーイング（変数の隠蔽）を明確に表現できる
- トランスパイル時に一意な名前を生成できる

### 2. スコープ識別子の実装詳細はカプセル化する

スコープ識別子の具体的な形式や出力方法は、内部実装の詳細として扱います。

**現状の実装**:
```ocaml
(* 内部実装の例 *)
let top_level_scope_id = ""
let make_var name scope_id = (name, scope_id)
```

**出力形式**:
- トップレベルスコープ: `name`（識別子なし）
- ローカルスコープ: `name__N`（`__` + 数値）
  - 例: `x__1`, `fact__2`

**カプセル化の理由**:
- 識別子の出力形式は今後変更される可能性がある
- 外部からは`var`型とコンストラクタ関数のみを公開する
- トランスパイラの内部実装に依存させない

### 3. スコープの動作要件

スコープシステムは以下の動作を保証する必要があります。

#### 3.1 スコープ外の変数は参照できない

```clojure
; NG: let式の外からxを参照
(def (bad-example)
  (let (x 10)
    x)
  x)  ; エラー: xはスコープ外
```

**実装上の対応**:
- 各変数は定義されたスコープ識別子を持つ
- 異なるスコープ識別子を持つ変数は別物として扱われる

#### 3.2 シャドーイング（変数の隠蔽）

内側のスコープで同名の変数が定義されると、外側の変数は隠されます。

```clojure
(def (example x)
  (let (x 10)      ; 引数xを隠す
    x)             ; 10を返す（let内のx）
  x)               ; 引数のxを参照
```

**実装上の対応**:
- シャドーイングは**新しいvarを作成する**ことで表現
- 同じ名前でも異なるスコープ識別子を持つため、自動的に別の変数として扱われる

```ocaml
(* 例: シャドーイングの表現 *)
let arg_x = make_var "x" "__1" in      (* 引数のx *)
let let_x = make_var "x" "__2" in      (* let束縛のx *)
(* arg_x と let_x は異なるvar値 *)
```

#### 3.3 トップレベルスコープ

組み込み演算子やグローバル関数は、トップレベルスコープに属します。

```clojure
(+ 1 2)              ; + はトップレベル
(def (add x y)
  (+ x y))           ; 関数内からもトップレベルの + を参照可能
```

**実装上の対応**:
- トップレベルスコープには特別な識別子を使用（現在は空文字列 `""`）
- トップレベルの変数はどのスコープからでも参照可能

```ocaml
let builtin_plus = make_var "+" top_level_scope_id
(* 出力: "+" （識別子サフィックスなし） *)
```

#### 3.4 ローカルスコープ

以下の構文はローカルスコープを導入します:

- **関数の引数**
  ```clojure
  (def (add x y)     ; x, y はこの関数のローカルスコープ
    (+ x y))
  ```

- **let束縛**
  ```clojure
  (let (x 10       ; x はlet式のスコープ
        y 20)      ; y はlet式のスコープ
    (+ x y))
  ```

- **matchパターン**
  ```clojure
  (match lst
    [] 0
    ; x, xs はこのケースのスコープ
    (x :: xs) (+ x (sum xs)))
  ```

**実装上の対応**:
- 各ローカルスコープには一意な識別子が割り当てられる
- ネストしたスコープでは、それぞれ異なる識別子を使用

## 実装の影響

**現在の設計**:
```ocaml
(* var値で判断する場合 *)
let rec contains_rec_call (var : var) expr =
  match expr with
  | Sym v -> v = var  (* タプルの等価性比較のみ *)
  | Fn (args, body) ->
    (* シャドーイング時は異なるvarが使われるため、自動的に区別される *)
    contains_rec_call var body
```

シャドーイングは**AST構築時に新しいvarを作成する責任**として扱われます。

### AST構築者の責任

AST（抽象構文木）を構築する側は、以下の責任を持ちます:

1. **スコープごとに新しい識別子を生成する**
   ```ocaml
   let scope_counter = ref 0
   let v name =
     incr scope_counter;
     make_var name ("__" ^ string_of_int !scope_counter)
   ```

2. **シャドーイング時は新しいvarを作成する**
   ```ocaml
   let outer_x = v "x" in    (* ("x", "__1") *)
   let inner_x = v "x" in    (* ("x", "__2") *)
   (* 異なるvarなので自動的に区別される *)
   ```

3. **同じスコープ内では同じvarを再利用する**
   ```ocaml
   let x = v "x" in
   Let ([Var x, Int 10], FnAp [Sym x; Int 5])
   (* 同じxを参照 *)
   ```

### トランスパイラの責任

トランスパイラ（コード生成器）は、以下の責任を持ちます:

1. **varを一意な出力名に変換する**
   ```ocaml
   let to_unique_var ((name, scope_id) : var) : string =
     Printf.sprintf "%s%s" name scope_id
   (* ("x", "__1") -> "x__1" *)
   (* ("+", "") -> "+" *)
   ```

2. **出力形式の変更に対応する**
   - 将来的に識別子の形式が変更されても、この関数を修正するだけで対応可能
