# トランスパイル処理の流れ（表層AST→コアAST→型検査）

パースから型検査、トランスパイルまでの処理の流れを整理する。
ここでの主眼は「表層AST（ユーザーの書き味に近い構文）」と「コアAST（型検査・コード生成の入力として扱う正規形）」を分け、
トップレベルを順序依存で走査しながら環境を更新しつつ、必要な脱糖を行ってコアAST化すること。

## ステータス

IN DISCUSSION

- 2026-03-21 開始

## 背景

現状の `lisp_ast`（PoC）は、関数定義の糖衣や中置など、表層構文に寄った表現を含む。
そのまま型検査器やトランスパイラで表層構文を理解し始めると、
Fixity規則（left/right/non）や関数定義糖衣の扱いが複数箇所に散らばりやすい。

このため、

- パース結果はまず表層ASTとして保持し
- トップレベル走査中に環境（FixityEnv / TypeEnv）を更新しながら
- `def` / `expr` については「コアAST化（必要な脱糖を含む）」を行って
- 型検査はコアASTに対して実施する

という流れを基本線としておく。

## 決定事項（暫定）

- トップレベルの宣言（`infix` と型宣言）は順序依存で後続にのみ有効
- 型検査器は中置演算や中置連鎖を直接は型検査しない（表層構文の責務を持たない）
- 現状の `lisp_ast` は表層ASTとみなし、型検査・トランスパイルの入力として別にコアASTを設ける
- コアAST化の中に「脱糖」を含める（現状の脱糖対象は中置演算と関数定義糖衣）

## 用語

- **表層AST**: ユーザーの構文に近いAST。糖衣構文を残してよい。
- **コアAST**: 型検査とトランスパイルが入力として扱うAST。正規形（糖衣を除去）に寄せる。
- **脱糖**: 表層ASTの表現をコアASTの表現に落とす変換。

## トランスパイル処理の流れ（暫定）

### 1) パース

- 入力（Lispテキスト）をパースし、トップレベルのフォーム列を得る
- この時点では、`infix` 宣言や型宣言の順序依存を尊重するため、
  「後で環境を参照して意味が決まる形」を無理に確定させない

### 2) 表層AST（フォーム列）

- パース結果を表層ASTとして保持する
- 表層ASTは糖衣構文を含んでよい
  - 例: 関数定義糖衣（`Func` のような束縛形）
  - 例: 中置・中置連鎖（FixityEnvで解釈される）

### 3) トップレベル走査（順序依存）

トップレベルを先頭から見ていく。

- 宣言が来たら環境を更新
  - `infix` 宣言 → FixityEnv 更新
  - 型宣言 → TypeEnv 更新
- `def` / `expr` が来たら、次を行う
  1. **コアAST化（脱糖を含む）**
     - その時点の FixityEnv を参照して、中置・中置連鎖を正規形へ変換
     - 関数定義糖衣（`Func`）を正規形へ変換
  2. **型検査**
     - その時点の TypeEnv を参照して、コアASTを型検査
  3. **環境の拡張**
     - `def` が導入する束縛を TypeEnv に反映し、後続の型検査に使う

### 4) トランスパイル

- 型検査済み（または型情報が揃った）コアASTを入力としてコード生成する
- トランスパイラは表層構文（中置連鎖や `Func`）を解釈しない前提とする

## コアAST化で行う脱糖（現状の対象）

### 1) 中置演算・中置連鎖

- FixityEnv（left/right/non）に従って、二項演算のネストによる正規形へ変換する
- 変換は「そのフォームを処理する時点のFixityEnv」を参照して行う（順序依存）

### 2) 関数定義糖衣（`Func` 束縛）

表層ASTで関数束縛を `Func (f, params, ret_ty)` のように持つ場合、
コアASTでは「値束縛（Val） + 無名関数（Fn）」へ落とすことができる。

- `Func (f, params, ret_ty), body`
  - ⟶ `Val (Bind f, ...), Fn (params, ret_ty, body)`

この変換により、コアASTにおける束縛の形を `Val` のみに正規化しやすくなる。

## コアASTを設ける動機（性質）

- 型検査器を小さく保てる
  - FixityEnvや中置連鎖の規則を型検査器が抱えない
  - `Func` のような糖衣構文を型検査器が抱えない
- 脱糖規則の一元化
  - 表層→コアの変換に規則を集約し、「型検査」と「トランスパイル」で二重実装になりにくい
- 仕様の整合性が取りやすい
  - 「型検査で受理した表現」と「トランスパイル時の解釈」が乖離しにくい

## 未決定事項

- コアASTに残す構文の最小集合（例: `LetRec` 相当が必要か）
- 再帰（自己再帰/相互再帰）の扱いと、コアASTでの表現
- トップレベル型宣言の意味（注釈として一致確認 vs 宣言型を前提にチェック）
- 型宣言のみ（定義なし）を将来許すか（import/モジュールを見越した interface 的扱い）
