## 実装の方向性

型システムの導入については、「型環境と基底の型をもとに、式の型を判断する」という方向性の型付け判断を行う。

型検査のアルゴリズムを構築するうえでの基本的なアプローチは下記

- 一旦単純な型から考える。int, bool, function, listから始める。
- 型環境はタプル(変数, 型)のリストとし、スコープについてはいったん考えない。あとで考える
- 初期値の型環境を考える(`+`, `<`など)
- 初期値の型環境をもとに、単純な式について、その束縛先の変数の型を導く
- 関数はカリー化。例えば、 `+` は次のような型である
  - `("+", Fn (Int, Fn (Int, Int)))`

## モナドとしての型検査器

モナドとしての型検査器を実装したい。

### 一旦の実装方針: Reader + Validation

まずは次を優先して実装する。

- 型環境の参照（スコープを含む）を扱う
- エラーの表現を整理し、必要な箇所では複数のエラーを蓄積できるようにする

この段階では、型検査を「型環境を読み取り、失敗する可能性がある計算」として表現する。

- Reader（環境の読み取り）: `env -> ...`
- Result（失敗）: `... -> ('a, type_check_error) result`
- Validation（失敗蓄積）: `... -> ('a, type_check_error list) result`

型環境は永続データ構造（`(var * type) list` など）として扱い、「更新」は破壊的変更ではなく拡張（先頭に束縛を追加）とする。

### Readerモナドの簡単な説明（`local` がスコープを表す）

Readerモナドは「共通のコンテキスト（ここでは型環境）を読み取る計算」を表す。

- 基本形: `type 'a t = env -> 'a`
- 型検査では失敗があるため: `type 'a t = env -> ('a, type_check_error) result` のようにする

ここで重要なのが `local` で、これは「計算を走らせる間だけ型環境を差し替える（拡張する）」ために使う。

- `local : (env -> env) -> 'a t -> 'a t`
- `let` の本質は「束縛式を現在の環境で検査し、その結果で環境を拡張して body を検査する」ことなので、`local` で自然に表現できる

例: `let x = e1 in e2` は、`e1` を検査して型 `t1` を得た後、`local (extend x t1) (judge e2)` のような形で表せる。

### Validation（失敗蓄積）の位置づけ

`Result` の `bind`（`let*`）で進めるモナド的な書き方は、基本的に fail-fast（最初の失敗で止まる）になる。
一方で、たとえば `match` の各分岐やリスト要素など「互いに独立に検査して、エラーをまとめたい」箇所では、Applicativeとしての Validation を使ってエラーを蓄積する。

### 将来の拡張（Stateが必要になる場面）

型変数の導入や単一化（unify）を行う段階では、型推論の進行状態（freshな型変数の発行、代入（substitution）、制約など）を保持したくなる。
この“推論の状態”はReaderの責務ではないため、必要になったタイミングで State（または状態の明示的な引き回し）を追加する。
